# Нагрузочные испытания Kafka и NATS

Для брокерных сценариев используется отдельная утилита `cmd/brokerstress`. Она запускается внутри контейнера `orders`, использует рабочие модули Kafka/NATS и обращается к тем же экземплярам `policy-server`, что и прикладные службы.

## Измеряемые режимы

`publish`:

```text
авторизация публикации -> HMAC-подпись -> подтверждённая публикация брокером
```

`e2e`:

```text
авторизация публикации
-> HMAC-подпись
-> публикация
-> доставка
-> проверка подписи
-> авторизация обработки
-> прикладной обработчик
```

`consume`:

```text
непосредственная публикация подписанного тестового сообщения
-> доставка
-> проверка подписи
-> авторизация обработки
-> прикладной обработчик
```

В режиме `consume` авторизация публикации намеренно исключена, чтобы отдельно оценить путь потребления. Для NATS результат включает сетевую доставку, поскольку обычный NATS не хранит сообщения без подписчика.

## Полный набор

Сначала собрать и запустить стенд:

```bash
make -C deploy up
```

Затем выполнить испытания:

```bash
make -C deploy broker-stress
```

По умолчанию выполняются:

- Kafka publish: 50000 сообщений, параллелизм 10/50/100;
- NATS publish: 50000 сообщений, параллелизм 10/50/100;
- Kafka/NATS end-to-end: 50000 сообщений, параллелизм 10/50/100;
- Kafka/NATS consume для валидной подписи, неверной подписи и запрета политики;
- повтор publish/end-to-end с отключённым кэшем.

Параметры:

```bash
make -C deploy broker-stress \
  BROKER_STRESS_N=50000 \
  BROKER_CONSUME_N=50000 \
  BROKER_STRESS_CONCURRENCIES=10,50,100
```

Короткая проверка:

```bash
make -C deploy broker-stress-smoke
```

## Переключение реплик под нагрузкой

```bash
make -C deploy broker-failover-stress
```

Сценарий запускает непрерывную публикацию, затем последовательно:

1. останавливает `policy-server-1`;
2. останавливает `policy-server-2`;
3. останавливает `policy-server-3`;
4. восстанавливает все реплики.

При доступности одной или двух реплик операции должны продолжаться. При остановке всех реплик и `FailOpen=false` ожидается запрет при отказе.

Короткая проверка:

```bash
make -C deploy broker-failover-stress-smoke
```

## Результаты

Каждый запуск создаёт каталог:

```text
results/broker-stress/<timestamp>/
```

или:

```text
results/broker-failover-stress/<timestamp>/
```

В каталоге находятся:

- отдельный JSON для каждого прогона;
- журнал выполнения;
- `summary.md` с общей таблицей.

JSON содержит:

- `avg_ms`;
- `p50_ms`;
- `p95_ms`;
- `p99_ms`;
- `throughput`;
- `success`;
- `errors`;
- число разрешённых, запрещённых и заблокированных операций;
- число вызовов прикладного обработчика.

Для сценариев `invalid-signature` и `denied` число вызовов обработчика должно быть равно нулю.
